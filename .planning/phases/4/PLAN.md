# Phase 4: AI Administrative Reduction — Bounded AI Drafting

## Goal
Implement AI-assisted user story drafting, decision summaries, and story-point estimation from cycle-time data. Strict guardrails — no hallucination tolerance, bounded roles, and safe execution via the existing OpenClaw/executor layer.

## Why This Phase
KeepUp.txt: "Developers spend too much time trying to write user stories than actually writing the code." This phase targets the administrative bloat that consumes developer time, using AI to draft (never auto-commit) administrative artifacts.

---

## Wave 1: AI Drafting Services (Tasks 4.1–4.3)

### Task 4.1: User Story Drafter
**Files**:
- `apps/worker/src/services/ai-recommendation-drafts.ts` (enhance)
- `apps/api/src/routes/inference.ts` (add endpoint)
**What**:
- Given a Slack thread or Jira issue description, draft a user story with: title, acceptance criteria, story points estimate, linked context
- Story points estimated from historical cycle-time data for similar issue types
- Output always marked as "AI Draft — review before use"
- Expose via `POST /api/inference/draft-story` for dashboard use
**Acceptance**:
- AI generates user story from thread context with acceptance criteria
- Story point estimate derived from historical cycle-time data (not guessed)
- All output labeled as draft requiring human review

### Task 4.2: Decision Summary Generator
**Files**:
- `apps/worker/src/services/ai-recommendation-drafts.ts` (add function)
- `apps/api/src/services/inference-engine.ts` (add `generateDecisionSummary()`)
**What**:
- Given a ledger commit with evidence links, generate a structured decision summary: context, decision, rationale, alternatives considered, impact
- Format for Confluence/Notion export (markdown with headings)
- Include "Generated by FlowGuard AI — verify accuracy" watermark
**Acceptance**:
- Decision summary generated from ledger commit + evidence
- Output includes structured sections (context, decision, rationale)
- All output watermarked as AI-generated

### Task 4.3: Sprint Estimation Assistant
**Files**:
- `apps/worker/src/services/sprint-retro.ts` (enhance)
- `apps/api/src/routes/inference.ts` (add estimation endpoint)
**What**:
- Analyze historical cycle-time data per issue type, team, and complexity
- When a new Jira issue is created, suggest story points based on: similar past issues, team velocity, complexity indicators
- Show confidence interval: "3-5 points (based on 12 similar issues, median 4)"
- Feed actual vs estimated data back to improve future estimates
**Acceptance**:
- Story point suggestions include confidence interval and comparable issues
- Estimates improve over time as more data accumulates
- Estimation accuracy tracked and visible in metrics

## Wave 2: Guardrails & Safety (Tasks 4.4–4.5)

### Task 4.4: AI Output Validation Layer
**Files**:
- `apps/worker/src/services/ai-orchestrator.ts` (enhance)
- New: `apps/worker/src/services/ai-guardrails.ts`
**What**:
- All AI outputs pass through validation: Zod schema check, content safety check, hallucination detection (entity names must exist in DB)
- Confidence scoring: AI self-rates confidence, low-confidence outputs get additional warning
- Rate limiting: max AI calls per company per hour to control costs
- Audit log: every AI call logged with input hash, output, model used, latency
**Acceptance**:
- AI outputs referencing non-existent entities are flagged
- Low-confidence outputs get additional "Low confidence — verify carefully" warning
- AI call audit log queryable by company/date

## Success Criteria
1. AI drafts user stories from Slack threads with acceptance criteria and estimated story points
2. Decision summaries auto-generated from ledger commits
3. Sprint estimation uses historical data with confidence intervals
4. All AI outputs validated, watermarked, and auditable
5. Acceptance rate > 60% for AI drafts within 30 days of deployment

## Risks
- **Hallucination**: AI may generate plausible but incorrect details. Mitigation: entity validation + "draft" labeling + human review gate.
- **Cost**: AI calls for every thread/issue could be expensive. Mitigation: rate limiting + caching + batch processing during off-hours.
- **Trust**: Developers may dismiss AI suggestions without reading. Mitigation: start with high-quality targeted suggestions (not everything), track acceptance rate.
